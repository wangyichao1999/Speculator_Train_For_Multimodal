# 草稿模型数据集生成修改说明

本文汇总本次为 **Eagle3 草稿模型训练数据生成** 所做的代码修改，覆盖以下文件：

- `generated_data/generate_training_data.py`
- `speculators/src/speculators/data_generation/vllm_hidden_states_generator.py`
- `speculators/src/speculators/config.py`

目标是让流程能够基于 `generated_data/training_data.json` 正确生成**多模态图文场景**的 hidden states 训练数据，并兼容当前本地 vLLM 版本。

---

## 1. 总体问题背景

原始流程存在两类核心问题：

1. **样本绑定逻辑错误**
   - 旧逻辑主要依赖 `oss_urls.txt` 和 URL 去匹配 teacher response。
   - 但当前真实数据源 `training_data.json` 里提供的是：
     - `image_paths`
     - `prompt`
     - `response`
   - 并没有稳定可直接使用的 `image_url` 字段。

2. **多模态请求没有真正进入 vLLM Request**
   - `generate_training_data.py` 已经构造了：
     - `prompt_token_ids`
     - `mm_features`
   - 但 `VllmHiddenStatesGenerator.generate()` 原先只按纯文本 token 序列处理，创建 `Request(...)` 时没有传递 `mm_features`。
   - 结果就是图像信息在 hidden state 抽取阶段被丢失。

此外，运行中还暴露出多处**本地 vLLM API 漂移兼容性问题**，需要逐项适配。

---

## 2. `generate_training_data.py` 的修改

文件：`/data/wyc/step1ot_eagle3_0403/generated_data/generate_training_data.py`

### 2.1 改为以 `training_data.json` 作为主样本源

原先流程偏向按 URL 驱动；现在改为直接从 `training_data.json` 读取样本，并以每条样本自带的内容为准：

- `image_paths[0]`
- `prompt`
- `response`

新增辅助函数：

#### `_read_json_list(...)`
用于读取 JSON 列表并做基本校验。

#### `_candidate_image_paths(raw_image_path, image_dir)`
用于从样本图片路径推导多个候选本地路径，支持：

- 直接使用 JSON 中原始路径
- 使用 `output_dir/images/<basename>`
- 将 `images_copy/` 映射为 `images/`

#### `_resolve_sample_image_path(sample, image_dir)`
用于从样本中解析可用本地图片路径。

#### `load_normalized_samples(...)`
这是新的主入口之一，负责：

- 读取 `training_data.json`
- 为每条样本归一化出：
  - `sample_index`
  - `prompt`
  - `response`
  - `image_path`
  - `raw_image_path`
  - `image_basename`
  - `fallback_url`
- 区分：
  - `valid_samples`
  - `missing_image_samples`

### 2.2 修复图片路径解析与下载兜底逻辑

现在的策略是：

1. **优先直接使用 `training_data.json` 中的本地图片路径**
2. 若该路径不存在，则尝试：
   - `output_dir/images/<basename>`
   - `images_copy -> images` 目录映射
3. 仅当本地路径不可用时，才回退到 `oss_urls.txt` 下载
4. 下载时优先保留 JSON 中对应图片的 basename，而不是只靠 URL 的 md5 命名

对应修改点：

### 2.3 扩展 `download_images(...)`

函数签名调整为：

```python
def download_images(urls: list[str], output_dir: str, preferred_filenames: list[str] | None = None):
```

新增能力：

- 支持外部传入 `preferred_filenames`
- 如果 JSON 中已有 basename，则优先用该文件名保存下载结果
- 避免 URL 命名体系和 `training_data.json` 命名体系脱节

### 2.4 主循环改为按归一化样本迭代

不再使用类似 URL->local_path 的驱动方式，而是直接遍历 `normalized_samples`：

```python
for sample in normalized_samples:
    local_path = sample["image_path"]
    sample_idx = sample["sample_index"]
    response_text = str(sample.get("response", "") or "").strip()
    prompt_text = str(sample.get("prompt", args.prompt) or args.prompt)
```

这样每条样本天然绑定：

- 图片路径
- prompt
- response

不会再出现“通过 URL 回查 teacher response”导致的错位问题。

### 2.5 继续复用现有多模态处理链

没有重写视觉预处理，而是继续使用已有链路：

- `OfficialStep3VLProcessor`
- `tokenize_multimodal_conversation(...)`

其中 `tokenize_multimodal_conversation(...)` 仍会输出：

- `input_ids`
- `loss_mask`
- `mm_features`
- `debug_info`

并且 `OfficialStep3VLProcessor.apply(...)` 通过 vLLM 官方多模态 processor 构造 `MultiModalFeatureSpec`。

### 2.6 保留批量请求结构，交给 generator 消费

当前批量调用保留如下结构：

```python
batch_requests = [
    {
        "prompt_token_ids": token_ids,
        "mm_features": mm_features,
    }
    for token_ids, mm_features in zip(batch_token_ids, batch_mm_features)
]
results = generator.generate(batch_requests)
```

这里脚本侧已经具备多模态请求数据，关键改动是在 generator 侧让它真正消费这些字段。

## 3. `vllm_hidden_states_generator.py` 的修改

文件：`/data/wyc/step1ot_eagle3_0403/speculators/src/speculators/data_generation/vllm_hidden_states_generator.py`

### 3.1 扩展 `generate()` 的输入类型

原先只支持：

- `list[list[int]]`
- `torch.Tensor`

现在扩展支持：

- `list[dict[str, Any]]`

每个请求项可以包含：

- `prompt_token_ids`
- `mm_features`（可选）

新增方法：

```python
def _normalize_requests(
    self,
    token_ids: list[list[int]] | list[dict[str, Any]] | torch.Tensor,
) -> list[dict[str, Any]]:
```

作用：

- 统一纯文本和多模态两种输入格式
- 将每条请求归一化为：

```python
{
    "prompt_token_ids": ids_list,
    "mm_features": mm_features,
}
```

### 3.2 在 `Request(...)` 中透传 `mm_features`

这是本次最关键改动之一。

原先 `Request(...)` 只带文本 token；现在改为：

```python
req = Request(
    request_id=req_id,
    prompt_token_ids=ids_list,
    sampling_params=SamplingParams(
        max_tokens=MAX_DECODE_TOKENS, temperature=SAMPLING_TEMPERATURE
    ),
    pooling_params=None,
    eos_token_id=self.tokenizer.eos_token_id,
    arrival_time=INITIAL_ARRIVAL_TIME,
    mm_features=request_data["mm_features"],
    block_hasher=self.block_hasher,
)
```

这样图像特征就会真正进入 vLLM V1 request 流程，支撑多模态 prefill hidden state 抓取。

### 3.3 保持 prefill-only 抓取逻辑主体不变

以下主逻辑没有被重写，只做兼容性扩展：

- scheduler.schedule()
- `_set_request_metadata`
- `execute_model(...)`
- `_get_captured_states`
- 返回结果的输入顺序映射

因此整体改动面尽量控制在“支持多模态请求 + 兼容当前 vLLM API”。

## 4. `speculators/config.py` 的修改

文件：`/data/wyc/step1ot_eagle3_0403/speculators/src/speculators/config.py`

### 4.1 为 `version("speculators")` 增加源码运行兜底

原先代码直接调用：

```python
version("speculators")
```

但当前环境是源码树运行，不一定安装了 package metadata，因此会报：

```python
PackageNotFoundError
```

现在修改为：

```python
from importlib.metadata import PackageNotFoundError, version

try:
    SPECULATORS_VERSION = version("speculators")
except PackageNotFoundError:
    SPECULATORS_VERSION = "0.0.0+local"
```

并在字段中使用：

```python
speculators_version: str = Field(
    default=SPECULATORS_VERSION,
    description="Version of the speculators library",
)
```

作用：

- 在已安装场景下继续读取真实版本
- 在源码运行场景下给出稳定默认值
- 不改变业务逻辑，只修复初始化失败问题

## 5. 为当前本地 vLLM 做的兼容性修复

除了多模态主逻辑，还修复了多处与本地 vLLM 版本不一致导致的问题。

### 5.1 `get_hash_fn_by_name` 导入路径兼容

原代码导入路径不适配本地环境，现改为：

```python
from vllm.utils import get_hash_fn_by_name
```

这是导入路径修正，不改变函数语义。

### 5.2 `get_kv_cache_config_from_groups(...)` 补充 `kv_cache_specs`

本地 vLLM 的函数签名要求传入 `kv_cache_specs`，因此调整为：

```python
kv_cache_config = get_kv_cache_config_from_groups(
    vllm_config=self.vllm_config,
    kv_cache_groups=kv_cache_groups,
    kv_cache_specs=kv_cache_spec,
    available_memory=cache_memory,
)
```

### 5.3 移除 `Scheduler(...)` 不再支持的 `block_size` 参数

本地 `Scheduler.__init__()` 不接受 `block_size`，因此移除该参数，保留：

```python
self.scheduler = Scheduler(
    vllm_config=self.vllm_config,
    kv_cache_config=kv_cache_config,
    structured_output_manager=structured_output_manager,
)
```

逻辑上 block size 仍然来自 `cache_config`，不是功能性变化。

### 5.4 用 `scheduler.update_from_output(...)` 替代已不存在的 `sample_tokens(...)`

本地 `MultiprocExecutor` 没有：

```python
self.executor.sample_tokens(model_output)
```

因此改为当前 vLLM V1 的执行路径：

```python
model_output = self.executor.execute_model(scheduler_output)
self.scheduler.update_from_output(scheduler_output, model_output)
```

这与本地 `EngineCore.step()` 的流程保持一致。

### 5.5 兼容 `_get_captured_states` 的 RPC 返回 list

运行中发现：

```python
self.executor.collective_rpc("_get_captured_states", unique_reply_rank=0)
```

在当前环境下可能返回 `list`，而不是原先假设的 `dict`。

因此新增兼容处理：

```python
request_states_dict = self.executor.collective_rpc(
    "_get_captured_states",
    unique_reply_rank=0,
)
if isinstance(request_states_dict, list):
    request_states_dict = next(
        (states for states in request_states_dict if states), None
    )

if not request_states_dict:
    raise RuntimeError("Failed to capture hidden states from worker")
```

这修复了：

```python
AttributeError: 'list' object has no attribute 'keys'
```

## 6. 修改后的数据流

修复后的主流程可以概括为：

1. 从 `training_data.json` 读取样本
2. 为每条样本解析本地图片路径
3. 必要时按 basename 下载缺失图片
4. 使用 `OfficialStep3VLProcessor` + `tokenize_multimodal_conversation(...)`
   生成：
   - `input_ids`
   - `loss_mask`
   - `mm_features`
5. 将请求组织为：

```python
{
    "prompt_token_ids": token_ids,
    "mm_features": mm_features,
}
```

6. `VllmHiddenStatesGenerator.generate()` 归一化这些请求
7. 在 `Request(...)` 中传入 `mm_features`
8. vLLM 执行多模态 prefill
9. 自定义 worker 抓取指定层 hidden states
10. 输出训练样本 `.pt`

## 7. 本次修改解决的核心问题

### 已解决

1. **teacher 数据与图片不再通过 URL 错配**
   - 以 `training_data.json` 为准
   - 样本级绑定 `image_path/prompt/response`

2. **多模态特征不再丢失**
   - `mm_features` 已从脚本传到 `Request(...)`

3. **源码树运行时不再因包版本元数据缺失而失败**
   - `speculators/config.py` 增加 fallback

4. **多处本地 vLLM API 兼容性问题已修复**
   - import path
   - kv cache helper 参数
   - scheduler 构造参数
   - model output 更新方式
   - captured states RPC 返回形态

### 仍需最终确认

虽然代码已经推进到更后面的执行阶段，但是否**完全端到端成功生成并落盘最终 hidden_states 数据**，还需要再跑一次完整流程确认最新补丁后的结果。
